Implementation guide.

Everything needed to run TinyBlog on ordinary PHP hosting: upload the files, create or configure the first admin, embed a feed or subscribe box, and keep the security path visible.

Docs 01

Quick start

Upload the project to a PHP 8 host with PDO SQLite, make data/ and uploads/ writable, then visit /admin to create the first admin account or log in with credentials configured in a private .env file. No default password ships with the app.

01

Upload tinyblog.php, tinyblog-widget.js, .htaccess, assets/, uploads/, and data/.htaccess to your site root or /blog subfolder.

02

Enable PHP 8.0 or newer, plus pdo_sqlite, sqlite3, and fileinfo when your host exposes extension toggles.

03

Open /admin, or first set TB_ADMIN_EMAIL and TB_ADMIN_PASSWORD in .env, then set the canonical URL, choose the site id, and add allowed widget origins.

04

Paste the widget snippet on an allowed page and publish sample posts if you want starter content.

Docs 02

Deployment

TinyBlog targets Namecheap-style cPanel hosting with Apache, PHP 8+, and SQLite. The app creates data/tinyblog.db on first run.

terminal 
chmod 755 data uploads
chmod 644 tinyblog.php tinyblog-widget.js .htaccess

# If PHP runs as another user and SQLite cannot create the DB:
chmod 775 data uploads

Keep data/ outside the public web root when your host allows it. If it must stay public, the included .htaccess denies direct downloads of databases, logs, and .env. Known login link: use /admin after editing credentials in the backend file.

.env 
TB_DB_PATH=/home/USER/private/tinyblog.db
TB_UPLOAD_DIR=/home/USER/public_html/blog/uploads
TB_LOG_FILE=/home/USER/private/tinyblog.log
TB_SESSION_TIMEOUT=1800
TB_LOGIN_RATE_LIMIT=10
TB_ADMIN_EMAIL=owner@example.com
TB_ADMIN_NAME=Owner
TB_ADMIN_PASSWORD=change-this-long-password

Docs 03

Embed snippets

The widget is dependency-free and reads JSON from your own TinyBlog endpoint. Put the script on a domain listed in Admin -> Settings -> Allowed widget origins.

feed widget 
<script
  src="https://blog.example.com/tinyblog-widget.js"
  data-tb-config='{"site":"store-1","endpoint":"https://blog.example.com/api","widgetType":"feed","maxItems":5}'>
</script>
manual init 
<div id="updates"></div>
<script src="https://blog.example.com/tinyblog-widget.js"></script>
<script>
  TinyBlogWidget.on("loaded", function (event) {
    console.log("TinyBlog loaded", event.type);
  });

  TinyBlogWidget.init({
    container: "#updates",
    endpoint: "https://blog.example.com/api",
    site: "store-1",
    widgetType: "feed",
    maxItems: 3,
    showExcerpt: true,
    accent: "#000000"
  });
</script>

Docs 04

Widget config

Configuration is passed as JSON in data-tb-config or directly to TinyBlogWidget.init().

KeyUse
endpointRequired API base URL, for example https://blog.example.com/api.
sitePublic site id. Defaults to store-1.
siteKeyOptional public key if API reads require it in Settings.
widgetTypefeed, post, or subscribe.
slugRequired for post widgets.
maxItemsFeed item count from 1 to 50.
themelight or dark; both stay monochrome.
accentCSS accent color. Defaults to #000000.
containerOptional CSS selector for manual placement.

Feed responses include items, page, and hasMore while preserving the older posts array. Long posts expose reading_minutes and reading_time.

Routes

The PHP file serves canonical pages, feeds, admin actions, and the JSON API from the same code path.

RoutePurpose
GET /Public archive listing.
GET /post/{slug}Canonical post page with SEO, social meta, reading time, and BlogPosting JSON-LD.
GET /tag/{tag}?page=2Paginated tag listing.
GET /search?q=term&page=2Paginated prepared-statement search with FTS5 fallback support.
GET /feed.xmlRSS feed.
GET /feed.jsonJSON Feed 1.1.
GET /api/posts?site=SITEWidget/API feed payload.
GET /api/posts/{slug}?site=SITESingle post API payload.
POST /api/subscribeRate-limited subscriber capture.
GET /adminLogin, dashboard, settings, posts, media, subscribers, and backup tools.

Docs 06

Security defaults

TinyBlog keeps the sensitive path small and inspectable: prepared SQL, escaped output, sanitized Markdown, browser-side widget sanitization, CSRF tokens, rate limits, password hashing, session timeout, image-only uploads, and exact-origin CORS allowlists.

Production checklist

  • Enable HTTPS before creating the first admin.
  • Use a unique admin password of at least 12 characters.
  • Restrict allowed widget origins to exact domains.
  • Keep .htaccess and uploads/.htaccess in place.
  • Back up data/tinyblog.db and uploads/ regularly.
  • Confirm raw Markdown HTML is escaped, not executed.

Auth flow caveats

  • Set TB_ADMIN_EMAIL and TB_ADMIN_PASSWORD_HASH before public launch when possible; otherwise the first visitor to /admin can create the first admin.
  • Prefer TB_ADMIN_PASSWORD_HASH over plaintext TB_ADMIN_PASSWORD; plaintext credentials are only for short setup or recovery windows and depend on .env staying private.
  • Remove TB_ADMIN_* values after bootstrap or recovery if you do not want the backend file to keep re-applying that admin password and role.
  • Force HTTPS and HSTS at the host so login credentials and session cookies never travel over plain HTTP.

Docs 07

Backups and moving hosts

Use Settings -> Backup for JSON export/import of posts, settings, confirmed subscribers, and media references. Copy uploads/ alongside the JSON when restoring elsewhere.

cron 
# Warm public feeds every hour
0 * * * * wget -q -O /dev/null https://blog.example.com/feed.xml
2 * * * * wget -q -O /dev/null https://blog.example.com/feed.json
5 * * * * wget -q -O /dev/null https://blog.example.com/sitemap.xml

# Daily SQLite backup
20 2 * * * cp /home/USER/public_html/blog/data/tinyblog.db /home/USER/backups/tinyblog-$(date +\%F).db

Docs 08

Testing

Run the smoke checks locally, then create the first admin in a browser because admin CSRF tokens are form-bound.

checks 
powershell -ExecutionPolicy Bypass -File tests\smoke.ps1
powershell -ExecutionPolicy Bypass -File tests\security_scan.ps1

php -l tinyblog.php
php tests/smoke.php
api probes 
curl -H "Origin: https://your-store.example" \
  "https://blog.example.com/api/posts?site=store-1&limit=3&page=1"

curl -X POST "https://blog.example.com/api/subscribe" \
  -H "Origin: https://your-store.example" \
  -H "Content-Type: application/json" \
  --data '{"site":"store-1","email":"reader@example.com"}'

Docs 09

How to scale

Move to Postgres when multiple authors, higher write volume, or large subscriber lists make SQLite locking noticeable. Move media to object storage when uploads outgrow shared-hosting disk. Move to a larger publishing stack only when you need webhooks, background jobs, team permissions, or multi-site tenancy.
